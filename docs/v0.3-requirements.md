# deep-report v0.3 — 需求文档

> Draft v0.3.0-rc1 · 2026-06-17 · 基于 v0.2.0 (1b52cd2) 现状演进

---

## 一、定位与基线

### 1.1 一句话目标

把 deep-report 从「单股季度报告生成器」升级为「**可复用、可对比、可追溯、可触发** 的财报分析基础设施」。

### 1.2 v0.2 现状（基线）

| 维度 | v0.2 状态 | 痛点 |
|------|----------|------|
| 流水线 | Fetcher→Analyzer→Writer 三层，Extract+Analyze 双 Pass | Extract 顺序执行，单股 ~60s |
| 文本提取 | PDF (pdfplumber) + HTML (XBRL降噪 89%→2%) | 扫描件不支持 |
| 采样 | 14锚点双语采样器 | 仅限核心字段 |
| 校验 | financial-sdk 3档阈值 | 美股 SDK 缺数据，无校验 |
| 叙事 | 5段递进式 + 6行业术语速查 | 风格固定，估值靠 LLM 知识 |
| 持久化 | 仅 `.ready` marker | KPI JSON 不落盘，无法复用 |
| 触发 | 全手动 CLI | 未与 morning-brief 集成 |
| 报告输出 | 飞书文档 | 无趋势图、无对比视图 |
| 测试 | 27 E2E + 114 单元 | 无黄金样本回归 |
| LLM | DeepSeek 主 + doubao fallback，温度0.3 硬编码 | 不可配置 |

### 1.3 v0.3 非目标

明确**不做**的事（避免 scope creep）：
- ❌ 重写 Extract/Analyze 的 LLM 逻辑（v3.6 prompt 已稳定）
- ❌ 接管 unified-downloader 的下载策略
- ❌ 替代 financial-sdk 的数据查询能力
- ❌ 自建 Web 后端（轻量用 Streamlit 即可）
- ❌ 多语言叙事（中英双语 prompt 不在 v0.3 范围）

## 二、核心主题（5 大模块）

按优先级 P0/P1/P2 划分，每个模块独立可交付、可验证、可回滚。

### 主题 A：数据持久化与复用 🔴 P0

**问题**：每次 analyze 都重跑 LLM Extract，token 浪费 + 不可重入 + 无法做趋势。

**交付**：
1. **KPI SQLite 持久化层**
   - 表结构：`kpi_extractions (id, code, period, market, kpi_json, extraction_ts, llm_model, prompt_version)`
   - 按 `(code, period, prompt_version)` 主键去重
   - 命中缓存 → 跳过 Extract Pass，秒级返回

2. **报告快照层**
   - `report_snapshots (id, code, period, narrative_md, validation_status, generated_ts)`
   - 同一份 KPI 多次 generate → 复用 KPI，仅重跑 Analyze Pass

3. **CLI 增量模式**
   ```bash
   deep-report analyze MNSO --period 2026Q1 --reuse-kpi   # 命中缓存
   deep-report analyze MNSO --period 2026Q1 --force        # 强制重跑
   ```

**验收**：
- 第二次 `analyze MNSO --period 2026Q1` 在 5s 内返回（命中缓存）
- 缓存命中率统计可查（`deep-report cache stats`）

---

### 主题 B：性能与并行化 🟡 P1

**问题**：5 期历史报告 × 60s/期 = 5min 等待；Extract 顺序执行是浪费。

**交付**：
1. **Extract 并行化**
   - `ThreadPoolExecutor(max_workers=3)`（与 LLM provider 限流对齐）
   - 失败单期不阻塞整体（标记 ERROR，继续后续）
   - 并发数可配置 `--extract-workers N`

2. **下载与 Extract 解耦**
   - `--download-only` / `--extract-only` 支持分步执行
   - 适配长时任务（避免 cron 1200s timeout 风险）

3. **超时分级**
   - Extract Pass：120s（保持）
   - Analyze Pass：180s（叙事生成需要更长）
   - 大型年报（>500 页）：自动切分多个 Analyze 子任务

4. **token 预算控制**
   - 单次 Extract 超 6K 输入 → 触发 14锚点再压缩
   - 全程记录 `tokens_in / tokens_out / cost_usd`

**验收**：
- 5 期历史：串行 300s → 并行 ≤ 120s
- 单期 Extract 失败不影响其他期

---

### 主题 C：估值/价格数据接入 🟡 P1

**问题**：ARCHITECTURE.md 第九节明确指出"无估值实时数据，估值段依赖LLM内置知识" → 准确性差、易幻觉。

**交付**：
1. **轻量估值数据层（val_metrics）**
   - 接入顺序：Longbridge CLI → 雪球 API → financial-sdk 缓存
   - 字段：当前价、PE-TTM、PB、PS、股息率、52周高低、市值
   - 数据时间戳必填（避免 LLM 误用旧数据）

2. **估值锚点改写**
   - 5段叙事中"估值锚点"段强制使用 val_metrics 数据
   - LLM prompt 注入：「以下为截至 {ts} 的真实估值数据，禁止编造」
   - 缺失字段 → 标注 `[N/A 数据缺失]`，不编造

3. **估值温度计（可选可视化）**
   - 当前 PE 在历史分位的位置（5年/3年/1年）
   - 与同业中位数对比

**验收**：
- 任何"PE = X.X"的数字都可在 `data/val_metrics.db` 找到对应行
- 缺失字段的 report 显示 `[N/A]` 而非 LLM 编造值

---

### 主题 D：行业模板与扩展性 🟢 P2

**问题**：v0.2 prompt 6 行业术语速查是硬编码在 markdown 里，新增行业要改 prompt 文件。

**交付**：
1. **行业模板 YAML 化**
   ```yaml
   # config/industries/consumer_retail.yaml
   industry: consumer_retail
   kpis_required: [same_store_sales, store_count, traffic, ticket_size]
   kpis_optional: [member_count, repurchase_rate]
   anti_patterns:
     - "B2B 收入不能与门店收入混算"
     - "海外门店数必须分地区披露"
   peer_set:
     - MNSO
     - PDD  # 海外对标
   ```

2. **运行时注入**
   - `--industry consumer_retail` 加载对应 YAML
   - 未指定 → 走默认"通用"模板
   - 模板可热加载（修改 YAML 无需重启）

3. **内置 8 行业模板**（覆盖 90% 场景）
   - consumer_retail / ecommerce / saas / semis / biotech / finance / industrial / energy
   - 每个模板附 peer_set（3-5 只对标股）

**验收**：
- `deep-report analyze PDD --industry ecommerce` 自动加载电商模板
- 新增行业 YAML 无需改代码

---

### 主题 E：自动触发与集成 🟢 P2

**问题**：v0.2 完全手动触发，无法融入 morning-brief / 飞书机器人 / cron 流水线。

**交付**：
1. **earnings event webhook**
   - morning-brief 检测到财报发布事件 → POST 到 `http://localhost:PORT/earnings`
   - deep-report daemon 监听 → 自动 analyze → 飞书推送
   - 提供 `--daemon` 模式启动 HTTP server

2. **飞书机器人命令**
   - 群消息 `@deep-report analyze MNSO 2026Q1` → 自动执行
   - 需要 feishu-cli im 消息接收集成
   - 简单实现：只读 chat history + 命令前缀匹配

3. **watchlist 模式**
   ```bash
   deep-report watch --config watchlist.yaml
   # watchlist.yaml: 每只股票 + 触发条件 + 推送目标
   ```

4. **统一 CLI 入口**
   ```bash
   deep-report analyze [CODE] [OPTIONS]    # 主命令
   deep-report cache stats/clear/list      # 缓存管理
   deep-report watch start/stop            # watchlist 守护
   deep-report daemon start/stop           # webhook 服务
   deep-report industry list/show          # 行业模板
   ```

**验收**：
- morning-brief 触发 → 60s 内 deep-report 自动开始 analyze
- 飞书群 @bot 命令成功执行并回报

## 三、可选主题（视时间决定）

### 主题 F：质量保证与回归 🟢 P2

- **黄金样本库**：`tests/golden/{code}_{period}.json`（KPI 期望值）
- **回归测试**：每次 prompt 改动 → 自动跑 5 只股票 × 3 期，diff KPI
- **报告打分**：5维度质量分（完整性/准确性/可读性/可操作性/可追溯），>85 分才推飞书
- **幻觉检测**：报告中的数字必须能在 KPI JSON 找到来源

### 主题 G：可视化 dashboard 🟢 P2

- **Streamlit 极简版**（不重造轮子）
- 视图：单股趋势图、同业对比表、watchlist 列表、报告历史
- 数据源：复用 SQLite 持久化层
- 移动端：响应式即可

### 主题 H：可观测性 🔵 P3

- 全链路 trace（fetcher → extract → analyze → validate → writer）
- token/cost dashboard
- 失败率、p50/p95 耗时

## 五、关键决策点（需用户拍板）

| # | 决策项 | 默认推荐 | 备选 |
|---|--------|---------|------|
| 1 | SQLite vs DuckDB | SQLite (本地轻量) | DuckDB (后续分析更方便) |
| 2 | 估值数据首选数据源 | Longbridge CLI | 雪球 API (但有 WAF 风险) |
| 3 | 自动触发是否需要 daemon | 是 (morning-brief 集成必需) | 仅 cron 轮询模式 |
| 4 | Dashboard 选型 | Streamlit | Gradio / Dash / 飞书多维表格 |
| 5 | v0.3 是否强制要求所有主题 | 否 (P0 + P1 即可发版) | 全做 (6 周延长到 8 周) |
| 6 | KPI 缓存是否带 TTL | 不带 (财报数据稳定) | 30 天 TTL (节省存储) |
